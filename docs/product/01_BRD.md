# BRD｜工业运维 Agent 业务需求文档

## 1. 项目背景

制造设备排障通常依赖现场经验、分散手册、历史工单和多个数据系统。资深人员的隐性经验难复用，且工业场景不接受无来源、无审计的模型建议。

《“人工智能+制造”专项行动实施意见》提出到 2027 年推动制造业大模型、工业智能体、工业高质量数据集和典型场景建设，并包含预测性维护方向。[国家数据局等八部门](https://www.nda.gov.cn/sjj/zwgk/zcfb/0112/20260107214358696030895_pc.html) 这说明方向具备产业基础，但不等于任意企业可立即落地。

## 2. 要解决的业务问题

1. 状态数据、历史记录、手册和专家经验分散，查找成本高。
2. 不同维修人员的检查顺序和风险意识不一致。
3. 通用聊天模型可能输出流畅但无法追溯的建议。
4. 数据平台、知识库和工单系统缺少自然语言任务协调层。
5. 新员工与跨班组人员难以快速获得适用型号的排障知识。

## 3. 业务目标与非目标

### 目标

- 聚合状态、历史、规则和知识，生成结构化排障草稿。
- 所有关键建议显示来源、版本或页码线索。
- 数据缺失、冲突或高风险时明确提示、降级或升级。
- 记录输入、工具调用、版本和人工反馈，支持复盘。

### 非目标

- 不控制真实设备、不改 PLC 或工艺参数。
- 不自动创建正式工单、采购或付款。
- 不用合成或公开数据宣传真实诊断准确率和经济收益。
- 没有真实数据和标签前，不宣称具备故障预测能力。

## 4. 价值假设与验证

| 对象 | 价值假设 | 验证方式 |
|---|---|---|
| 维修技术员 | 缩短资料查找与初步排查准备时间 | 对照人工流程记录耗时 |
| 可靠性工程师 | 提升证据完整性和复盘效率 | 专家盲评、引用覆盖率 |
| 维修主管 | 统一风险分级和审计过程 | 工单复盘、审计抽样 |
| 企业 | 减少重复排查，改善部分场景 MTTR | 真实试点前后对比 |

行业资料常提到成熟预测性维护项目可能减少停机、延长设备寿命，但这些是行业参考，不是本项目承诺。[McKinsey](https://www.mckinsey.com/capabilities/operations/our-insights/manufacturing-analytics-unleashes-productivity-and-profitability)

## 5. 利益相关方

| 角色 | 主要责任 |
|---|---|
| 业务负责人/厂长 | 试点范围、预算、收益判断 |
| 维修主管 | 流程、采纳、工单规范 |
| 设备/可靠性工程师 | 规则、知识、案例和结果复核 |
| 一线维修人员 | 现场输入、使用与反馈 |
| IT/OT 与安全团队 | 接口、网络、权限、日志和合规 |
| 产品与研发 | 交付、质量、迭代和运维 |

## 6. 指标与安全红线

北极星指标：每 100 次有效排障会话中，经专家确认“证据充分且可作为下一步核查依据”的草稿数。

安全红线：未经确认外部写操作为 0；自动设备控制为 0；高风险漏升级为 0；无来源关键建议为 0；把数据失败伪装为查询成功为 0。

## 7. 立项 Go/No-Go

进入真实试点前必须同时具备：明确设备族与负责人、合法数据/手册授权、领域和安全专家、人工流程基线、部署与权限方案、独立测试集、高风险清单，以及业务方对“只读 + 草稿 + 人工确认”边界的接受。

任一关键条件缺失时，项目保留为教学/演示原型，不进入生产验证。
