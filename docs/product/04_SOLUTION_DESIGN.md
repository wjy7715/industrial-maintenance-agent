# 项目实现方案｜工业运维 Agent 技术设计

## 1. 设计原则

1. 安全与权限由确定性策略执行，不能只依赖模型。
2. Agent 负责规划和汇总，事实必须来自受控工具。
3. 无证据时降级或拒答，证据优先于语言流畅度。
4. UI、编排、工具、数据、知识、安全、审计和模型适配各自单一职责。
5. 先本地和只读集成，后受控写入；模型可替换。

## 2. 目标架构

```text
Web / API / 企业消息入口（后续）
              │
       认证、会话与权限层
              │
       Orchestrator 总控 Agent
  ┌───────────┼───────────┐
  │           │           │
状态工具    历史工具    风险规则工具    知识检索工具
  │           │           │
遥测适配器  CMMS适配器  规则仓库       知识仓库/检索器
              │
   审计日志 / 策略引擎 / 可观测性
              │
      结构化草稿 + 证据 + 反馈
```

## 3. 模块边界

| 模块 | 职责 | 禁止事项 |
|---|---|---|
| Channel/UI | 输入、展示、确认 | 直接访问生产数据源 |
| Auth/Session | 身份、站点和设备权限 | 参与诊断推理 |
| Orchestrator | 规划、工具调用、汇总 | 伪造工具结果、覆盖高风险规则 |
| Tool Registry | 工具注册、Schema、超时、重试 | 保存业务结论 |
| Telemetry/History Adapter | 标准化读取状态和历史 | 修改设备/正式工单 |
| Risk Engine | 执行版本化风险规则 | 允许模型改规则 |
| Knowledge Retrieval | 按权限检索带引用知识 | 返回停用或错误型号知识 |
| Model Adapter | 可选模型/Hermes 调用 | 持有设备控制权限 |
| Policy/Audit | 权限、确认、日志、监控 | 被文档提示覆盖 |

## 4. Agent 协作

MVP 使用“一个总控 + 多个有严格输入输出契约的专员/工具”：状态专员只输出遥测事实；历史专员只输出事件；风险专员执行确定性规则；知识专员只输出带来源片段；综合器区分事实、推断和未知项。不要为了多 Agent 而引入自由协商；跨设备、工单和排班复杂任务再考虑图式并发工作流。

## 5. 数据与工具契约

核心请求至少包含：`request_id`、`site_id`、`equipment_id`、`symptom_text`、`observed_at` 和可选工况。

每个工具统一返回：`tool_name`、`tool_version`、`status(success|partial|empty|failed)`、`data`、`source`、`observed_at`、`quality(good|stale|unknown)`、`warnings`、`error`。

最终草稿至少包含：`risk_level`、`facts`、`candidate_causes`、`inspection_steps`、`recommended_actions`、`unknowns`、`conflicts`、`citations`、`tool_trace_id`、`requires_expert_review`。所有契约在运行时校验，校验失败不得包装成成功。

## 6. 编排逻辑

1. 校验身份、设备和输入。
2. 识别会改变风险结论的缺失信息并追问。
3. 并行调用状态、历史与知识工具；风险引擎读取标准化事实。
4. 检查数据新鲜度、单位、来源状态、设备型号和冲突。
5. 高风险先输出升级要求，再决定可展示的常规核查范围。
6. 使用模板或模型生成草稿，再由输出验证器检查引用、Schema 和禁止操作。
7. 保存审计记录和用户反馈。

## 7. 检索与知识方案

起步使用“设备型号/部件/版本/状态”的元数据过滤 + 关键词/BM25 检索；返回原文片段、页码和适用范围。只有独立评测证明语义表达导致关键词召回不足，且混合检索不会增加错误型号匹配时，再引入向量数据库。

RAG 内容一律视为不可信数据：文档不能改变权限；检索前先做站点/设备权限过滤；片段保留哈希与版本；输出后验证引用是否支持相邻建议。

## 8. 数据接入与部署路线

1. 演示：本地 JSON/CSV/SQLite、合成遥测、公开资料。
2. 试点：只读导入真实设备清单、历史工单与抽样遥测。
3. 在线：通过只读 API、数据库账户、OPC UA/MQTT 网关接入。
4. 集成：确认后生成 CMMS 工单草稿；不自动写入。
5. 扩展：实时流和预测模型必须经过独立数据/安全阶段门。

演示阶段可使用单机 Streamlit；试点阶段应采用 API、独立数据库、企业认证、HTTPS、集中日志和企业内网/受控网络部署。Agent 不直接连接 PLC；OT 数据通过工业网关或数据平台隔离。

## 9. 模型、Hermes 与安全

- `ModelAdapter` 统一模型输入输出、超时、成本、脱敏和版本记录。
- Hermes 可用作规划/综合实现之一，但不可绕过工具白名单、策略和输出验证。
- 模型不可用时回退到规则 + 模板基线。
- 外部模型调用必须先通过数据分级和脱敏策略。
- 写操作逐次确认；MVP 架构中不注册任何设备控制工具。

监控 request_id、trace_id、工具耗时、失败率、降级、证据缺失、危险反馈和越权尝试；知识、规则、提示和模型版本均可回滚。上游中断时展示最后成功数据时间，不能静默使用缓存冒充实时结果。
