# 架构说明

## 设计原则

1. Agent 只协调，不把业务规则藏在提示词中。
2. 工具只读且单一职责，便于测试和替换。
3. 知识、运行数据和风险规则分层。
4. Hermes/LLM 只做意图与表达增强，结构化方案是唯一权威结果。
5. 真实外部操作必须另建适配器和确认协议，本项目不提供控制工具。

## 数据流

`DiagnosisRequest` 进入 `MaintenanceOrchestrator` 后依次获得遥测、历史、风险和知识结果。总控将这些结果汇总成 `MaintenancePlan`，随后由 `SafetyPolicy` 添加不可绕过的警告。CLI、网页和 Hermes 只消费同一个结构化结果。

## 为什么不是简单 RAG

- 遥测和历史来自不同工具，不进入知识文本检索。
- 数值风险由明确规则计算，不靠大模型猜测。
- 输出包含完整工具轨迹。
- 高风险结果强制升级人工处理。
- 无知识命中时返回空措施，而不是让模型补写。
