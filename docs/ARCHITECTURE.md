# 架构说明

## 设计原则

1. Agent 只协调，不把业务规则藏在提示词中。
2. 工具只读且单一职责，便于测试和替换。
3. 知识、运行数据和风险规则分层。
4. Hermes/LLM 只做意图与表达增强，结构化方案是唯一权威结果。
5. 真实外部操作必须另建适配器和确认协议，本项目不提供控制工具。

## 数据流

`DiagnosisRequest` 进入 `MaintenanceOrchestrator` 后，通过统一 `ToolResult` 契约获得遥测、历史、风险和知识结果。每个结果包含 success/empty/failed、版本、耗时、质量、来源与错误。总控将事实、候选原因、未知项和冲突汇总成 `MaintenancePlan`，随后由 `SafetyPolicy` 添加不可绕过的警告。会话写入本地 SQLite 审计库，CLI、网页和 Hermes 只消费同一个结构化结果。

`evaluation/shadow.py` 只读取审计库，计算影子试点指标并输出结构化报告。它不修改诊断计划或知识库，也不把本地样例指标描述为真实工厂准确率。网页看板和 CLI 的 JSON 报告共享同一评测服务。

`TelemetryCsvRepository` 实现与仿真设备仓储相同的只读查询边界。总控通过依赖注入选择数据仓储，因此切换脱敏 CSV 时不改变风险、知识、安全和审计逻辑。适配器对文件规模、编码、模式、数值、重复主键和时间戳执行失败关闭校验；来源类型随工具结果写入证据。

`MaintenanceHistoryCsvRepository` 与遥测仓储独立注入。它按带时区的事件时间重放告警开启和关闭状态，并保存已完成维修的动作、结果、确认原因与验证时间。总控只把这些内容作为历史证据，不直接提升为当前候选原因。

任一非关键工具失败时，系统保留失败轨迹并降级输出，不把缺失数据伪装成成功。遥测入口失败或设备不存在时停止诊断。知识检索只返回 active 且适用于当前设备型号的内容。

遥测进入总控前必须通过设备类型对应的指标、单位和合理范围契约；风险规则同时校验标准单位。工具执行前由 `ToolPermissionRegistry` 做确定性授权：只读查询和本地计算自动允许，外部写入需要明确确认，设备控制及未注册工具拒绝。最终 `MaintenancePlan` 在审计前由独立验证器检查引用、安全升级、禁止动作和人工确认；失败时清空具体动作并标记安全阻断。

## 为什么不是简单 RAG

- 遥测和历史来自不同工具，不进入知识文本检索。
- 数值风险由明确规则计算，不靠大模型猜测。
- 输出包含完整工具轨迹。
- 高风险结果强制升级人工处理。
- 无知识命中时返回空措施，而不是让模型补写。

## 本地持久化

`data/runtime/assistant.db` 保存诊断会话和人工反馈，目录不提交到 Git。反馈不会自动发布知识或改变规则。生产试点前需替换为企业数据库、统一身份认证、角色权限和正式备份策略。
