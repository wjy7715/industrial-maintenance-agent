# 正规产品开发与发布流程

## 1. 版本目标与范围

当前目标是完成一个可演示、可测试、可审计的工业离心泵运维 Agent 本地版。完成标准以 PRD 的本地 P0/P1 能力、安全红线和交付物为准，不以代码数量或页面数量为准。

企业认证、真实 CMMS/EAM、OPC UA/MQTT、现场阈值和专家盲评依赖外部组织、数据和授权。未获得这些条件时，只交付适配接口、模拟/脱敏数据验证能力和明确的试点接入说明，不宣称生产可用。

## 2. 开发流程

### 阶段 A：需求冻结与差距分析

- 维护 BRD、MRD、PRD、技术方案和需求追踪矩阵。
- 为每项需求指定验收标准、测试证据和边界。
- 将新增想法放入待办，不在当前迭代中临时扩张范围。

退出条件：本地发布范围、Won't、依赖外部条件和优先级明确。

### 阶段 B：架构与数据设计

- 先定义领域模型、工具契约、数据 Schema、权限和失败行为。
- 数据源、模型、UI 与存储通过适配器隔离。
- 任何外部数据默认不可信；缺失、过期、冲突和失败必须可见。

退出条件：设计可追溯到需求，关键安全决策不依赖提示词。

### 阶段 C：增量实现

- 每次只实现一个可闭环的垂直切片。
- 先写或同步验收用例，再实现，再运行回归。
- 只在正确性、安全、性能或维护性有明确收益时重构。
- 使用本地 Git 提交保存安全检查点；产品冻结前不推送远程。

退出条件：本切片功能、错误处理、日志、测试和文档同步完成。

### 阶段 D：验证与安全评审

- 单元测试、契约测试、端到端测试、失败注入和性能基线分层执行。
- 建立独立盲测集和安全红线矩阵，测试集不得用于调优。
- 危险建议、越权、严重风险漏升级和伪造工具成功的容忍值为 0。

退出条件：自动化回归全部通过，已知限制记录完整，阻断级缺陷为 0。

### 阶段 E：发布准备与统一验收

- 冻结代码和数据 Schema，生成版本说明与变更日志。
- 完成安装、操作、故障排查、备份恢复、演示脚本和测试报告。
- 制作项目总结汇报 PPT，并进行逐页视觉检查。
- 用户统一验收后，再一次性推送 GitHub 并更新 PR。

退出条件：代码、文档、演示、PPT、测试证据一致，远程仓库无遗漏。

## 3. Definition of Ready

需求进入开发前必须具备：业务目标、范围、输入输出、数据来源、权限边界、异常行为、验收标准和依赖。如果其中一项会改变产品方向且无法从现有文档确定，才向用户提问。

## 4. Definition of Done

一项功能只有同时满足以下条件才算完成：

1. 正常、空结果、非法输入和依赖失败均有明确行为。
2. 权限、安全和数据来源边界可见，不伪造执行结果。
3. 自动化测试覆盖关键路径，并通过全量回归。
4. 审计、错误信息和结构化输出足以复现问题。
5. README、架构、数据契约、状态和需求矩阵同步更新。
6. 不遗留未说明的临时代码、密钥、真实敏感数据或构建产物。

## 5. Git 与发布策略

- 开发分支：`codex/safe-mvp-v1.1`，后续保持不变直到最终验收。
- 允许本地小提交，保证断电或上下文切换后可以恢复。
- 从 2026-07-21 起暂停远程推送；当前本地比远程领先 1 个提交。
- 最终推送前重新运行全量测试、编译、差异检查和秘密扫描。
- 推送后核对 PR 文件、提交、检查结果与文档链接。

## 6. 质量记录

每个阶段都在 `PROJECT_STATUS.md` 更新版本、已交付、限制和下一阶段。测试数量只是回归规模，不等于工业准确率。真实价值、MTTR 和危险建议率只能在有授权数据与专家评审的影子试点中报告。
